## Goal

Real SMS OTP login works on a real phone (test target: **+8613162908096**). Unlock engine is live — completing an activity unlocks the next one per product rules, with env-configurable end-of-Part timing. Server is fully stateless: all auth/session state in Redis or PG. Importer's media upload path (deferred from P1) is implemented so the Contract §2.4 operator workflow is complete end-to-end.

- [ ] Add `REDIS_URL=redis://localhost:6379` to `.env.development`, `.env.test`, `.env.production`.
- [ ] Move `scripts/redis-test.js` and `scripts/send-sms-aliyun.js` → `scripts/dev/`.

### §2. SMS lib rewrite (keep pop-core, port sibling logic)

**Decision:** keep `@alicloud/pop-core` + `@alicloud/credentials` (already installed). Port the sibling's *logic* without swapping SDKs.

- [ ] Rewrite `src/lib/aliyunSms.js`:
  - Module-scoped client cache (singleton).
  - `sendSms({phone, code, signName, templateCode})` returns `{success: bool, message: string, bizId?: string}` — **never throws**.
  - Parse Aliyun `Code === 'OK'`; log `bizId` for audit trail.
- [ ] `signName` / `templateCode` env-configurable via `ALIYUN_SMS_SIGN` / `ALIYUN_SMS_TEMPLATE`. Default to `北京自成长教育科技` / `SMS_504055089` (sibling's approved template — currently default is `SMS_485330444` which needs to change).

### §3. Auth service → Redis-backed

- [ ] Add `_normalizePhone()` helper — strips leading `+86` before `_assertPhone`. So both `+8613162908096` and `13162908096` land valid. All auth endpoints that accept `phone` normalize on entry.
- [ ] `captcha:{id}` → `redis.set(value, 'EX', 300)`. Delete on successful verify.
- [ ] `otp:{phone}` → `redis.set(JSON.stringify({code, attempts}), 'EX', 300)`. Increment attempts on wrong guess; lock after 5.
- [ ] `otp-rl:phone:{phone}` → `incr` + `expire 60`; reject with 429 if >1 per 60s.
- [ ] `otp-rl:ip:{ip}` → `incr` + `expire 3600`; reject with 429 if >10/hour per IP.
- [ ] Refresh tokens carry a `jti` (uuidv4) in payload. `logout` → `redis.sadd('jti-revoked', jti)` with TTL = remaining refresh lifetime. `refresh` → `SISMEMBER` check → 401 if revoked.
- [ ] Also persist active sessions to `user_sessions` PG table on login; flip `is_active=false` on logout (audit survives Redis TTLs).
- [ ] **Grep gate** — post-migration, zero hits in `src/` for: `this.captchaData`, `this.otpData`, `this.tokenData`, `captchaData: asValue`, `otpData: asValue`. Server is stateless.

### §6. Unlock engine

**Source of truth:** `user_progress` (existing). Day/Part/Course unlock state is **derived** from it, not stored separately. `unlock_logs` remains the append-only audit log.

- [ ] `prisma/schema.prisma`: add `unlocks_at DateTime?` to `user_progress` (null = immediately available if `status=UNLOCKED`; non-null = unlocks at that timestamp). `prisma db push` to apply.
- [ ] New `src/utils/tzHelper.js` — `nextBoundary(tz, hour)` returns next `hour:00` in `tz` as a UTC Date, using `Intl.DateTimeFormat`. No moment-timezone dep.
- [ ] New `src/services/unlockService.js`:
  - `computeUnlockedState(userId, courseId)` → `Map<activityId, {unlocked, unlocks_at}>`, derived from `user_progress` + hierarchy joins.
  - `recomputeAfterCompletion(tx, userId, activityId)` — called inside `sessionService.completeSession`'s existing transaction, right after gold award + user_progress upsert.
- [ ] Rules:
  - Sequential within Day: activity N+1 becomes UNLOCKED when N is COMPLETED.
  - Next Day unlocks when **all** activities in current Day are COMPLETED (all required — no `required` flag in schema, so every activity counts).
  - Next Part's Day 1 unlocks when previous Part fully cleared.
  - End-of-Part / end-of-arc: `unlocks_at = tzHelper.nextBoundary(UNLOCK_TZ, UNLOCK_HOUR)`. Env defaults `Asia/Shanghai` / `10`. **Env-configurable — not hardcoded.**
  - Lazy eval, no cron. Every course/part/activity GET runs `computeUnlockedState`; time-delayed rows flip unlocked on the first read after their `unlocks_at`.
- [ ] Response-shape changes — inject `{ unlocked: bool, unlocks_at: timestamp|null }` at course/part/day/activity granularity in:
  - `src/course/services/courseService.js`
  - `src/parts/services/PartsService.js`
  - `src/activities/services/ActivitiesService.js`
- [ ] `UNLOCK_BYPASS=true` stays as the demo escape hatch (short-circuits to all-unlocked). Default `false` in `.env.production` once §6 lands and tests are green.

### §7. Clean up loose ends from P4

## Risks

- [ ] **Aliyun SMS template approval lead time** — mitigated, reusing sibling's approved `SMS_504055089`.
- [ ] **Real Aliyun creds required for the live SMS test** — user populates `.env.local`. Don't commit.
- [ ] **Unlock engine complexity** — densest piece of the sprint. Budget a full day. Mitigated by reusing `user_progress` instead of introducing new tables.
- [ ] **10:00 Beijing assumption** — mitigated by making it env-configurable (`UNLOCK_TZ`, `UNLOCK_HOUR`). Client can tune without a code change.
- [ ] **Media upload failures mid-run** — idempotent via `media_processing_log` unique index; operator retries `upload:media` without re-running `import:pg`.
- [ ] **OSS prod bucket reachability** — P4 never tested actual downloads. Closed in §7 with `oss-smoke.js`.
- [ ] **Test redis container** — CI needs Docker available. `docker-compose.test.yml` pins the image and port.
