## Steps

### 1. Real SMS-based auth (Contract §2.2)

**Starting point on `origin/dev`**: all the endpoints exist and work functionally, but storage is in-memory. P5 §1 is essentially a **storage-layer rewrite**, not a from-scratch build.

- [ ] Port sibling project's `contenthubproductdoc/frontend/src/lib/sms.ts` over `src/lib/aliyunSms.js`. Sibling has better error handling and response validation. Keep `signName: "北京自成长教育科技"`, `templateCode: "SMS_504055089"` as default (also env-configurable).
- [ ] Migrate all captcha/OTP/refresh state to Redis in `src/auth/services/authService.js`:
  - [ ] `this.captchaData[id] = ...` → `redis.set('captcha:{id}', answer, 'EX', 300)`
  - [ ] `this.otpData[phone] = ...` → `redis.set('otp:{phone}', '{code,attempts}', 'EX', 300)`
  - [ ] Rate limits: `redis.incr('otp-rl:phone:{phone}')` + `EX 60`; `redis.incr('otp-rl:ip:{ip}')` + `EX 3600`.
  - [ ] Refresh tokens: `redis.sadd('jti-revoked', jti)` on logout, with TTL = refresh-token lifetime.
- [ ] `POST /api/v1/auth/otp` — already exists; add Redis-backed rate limiting.
- [ ] `POST /api/v1/auth/phone-login` — already exists; replace OTP lookup from in-memory to Redis.
- [ ] `POST /api/v1/auth/refresh` — already exists; add JTI revocation check.
- [ ] `POST /api/v1/auth/logout` — already exists as stateless; add JTI add-to-revocation-set.
- [ ] JWT secret in `.env.dev` (dev) / the operator's env file (prod). Rotation: swap env var + restart (documented, not automated).
- [ ] **Critical:** after this step, grep `this.captchaData`, `this.otpData`, `this.tokenData` — zero hits in `src/`. Server is stateless.
- [ ] Move `scripts/redis-test.js` and `scripts/send-sms-aliyun.js` to `scripts/dev/` (or delete if redundant with a real smoke test).

### 2. Unlock engine (Contract §2.5)

- [ ] New: `yyddp-api/src/services/unlockService.js`.
- [ ] Rules:
  - [ ] Sequential unlock within a Day (activity N+1 unlocks when N complete).
  - [ ] Next Day unlocks when current Day's required activities all complete.
  - [ ] Next Part's Day 1 unlocks when previous Part fully cleared.
  - [ ] End-of-Part / end-of-arc: unlocks at next calendar day 10:00 Beijing time (assumed — confirm with client).
  - [ ] Lazy evaluation, no cron. Every read recomputes.
- [ ] New tables in `prisma/schema.prisma`:
  - [ ] `user_unlocks` (user_id, entity_type, entity_id, unlocked_at, reason)
  - [ ] `user_unlock_audit` (append-only log)
- [ ] Surface `unlocked: bool` + `unlocks_at: timestamp | null` on every course/part/day/activity payload.
