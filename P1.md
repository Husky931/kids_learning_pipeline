# P1 — Import it into PG

**Maps to:** user step 2 "Importing it (into the pg)". Contract Appendix 2 §2.3. **Repo:** `yyddp-importer`. **Prev:** [P0](P0.md) | **Next:** [P2](P2.md) | **Index:** [roadmap.md](roadmap.md)

---

## Goal

Operator clicks "Import to PG (N changes)" in the admin UI; loader writes to dev PG. Re-running with the same SQLite produces zero diff. Soft-deletes work. No physical DELETEs ever.

## Steps

### 1. Loader core

- [x] New: `yyddp-importer/src/loader/load.ts`. (+ `ids.ts`, `types.ts`, `sanityChecks.ts`, `mediaInventory.ts`)
- [x] Input: `data.json` + `diff.json`. Per delta:
  - [x] `added` → INSERT
  - [x] `changed` → UPDATE (only the changed fields)
  - [x] `removed` → soft-delete: `UPDATE ... SET deleted = true` (Note: the Prisma schema has `deleted` but no `deleted_at` column — we update `updated_at` as the audit timestamp. If a `deleted_at` column is wanted later, it's a schema migration on `yyddp-api`.)
  - [x] `unchanged` → skip
- [x] **Order:** Course → Lesson → Part → Day → Activity → Interaction → matching sub-rows. Parent-first for inserts, child-first for soft-deletes.
- [x] One Prisma interactive transaction per course. Rollback on any error inside the course. (5-minute timeout configured in `load.ts`.)

### 2. Media upload (minimum viable)

**Scoped down to inventory-only for P1** — see `src/loader/mediaInventory.ts`. Walks every `/download/*` URL the import touches, cross-references `media_processing_log` in SQLite, reports `total / known / unknown` in the result panel. Full upload path remains P5.

- [ ] For activities whose media URL is referenced but not yet uploaded, upload local file to OSS at the canonical path. → **deferred to P5** (reported but not executed)
- [ ] Reuse `yyddp-importer/windmill/scripts/*` and/or the existing `upload_to_oss.ts` stub. → **deferred to P5**
- [ ] Record uploads in the `media_processing_log` table (already added by `scripts/migrateDb.ts`) so re-runs don't re-upload. → **deferred to P5** (we read the log, don't write it)
- [x] Not required at P1: full transcoding/watermarking. That's P5. We just ensure every URL resolves. → inventory surfaces unknown URLs in the UI so P5 has a work list.

### 3. Admin UI "Import" button

- [x] Extend `yyddp-importer/server/index.ts`: `POST /api/import` runs `load.ts` and streams progress (via NDJSON — one `ProgressEvent` per line; cleaner than SSE for a single POST round-trip). Routes live in `server/routes/import.ts`; also exposes `GET /api/import/status` for the UI to pre-flight.
- [x] Vue Diff tab gets "Import to PG (N changes)" button. (in `preview/src/components/DiffView.vue`)
- [x] Result panel: inserted / updated / soft-deleted / errors, one row per table. Also shows sanity checks + media inventory.
- [x] Button disabled unless `diff.json` has deltas AND `IMPORT_TARGET` is set. (Also: single-flight guard on the server rejects concurrent POSTs with 409.)

### 4. Safety rails

- [x] `IMPORT_TARGET=dev` is default. `IMPORT_TARGET=prod` requires:
  - [x] CLI flag `--yes-i-know`, OR
  - [x] UI checkbox "I am writing to production" + confirmation dialog (both enforced — button disabled until checkbox + `window.confirm()` on click).
- [x] All writes are soft-delete per Contract Appendix 2 §1. (Junction tables — `courses_parts`, `part_days`, `day_activities`, `activity_interactions` — are plumbing, not content; they're inserted/updated directly. The contract targets content tables.)
- [x] **Hard rule:** PG writes happen ONLY via the Import button (UI) or the equivalent CLI with explicit `--yes-i-know` flag. No scheduled job, no test harness, no background worker writes to PG. The operator click is the only write trigger. (All `prisma.*create|update|updateMany` calls in the importer tree live under `src/loader/`, invoked only by `scripts/importToPg.ts` or `server/routes/import.ts`.)

## Definition of Done

- [x] Click Import in admin UI → loader writes to dev PG. (Smoke-tested end-to-end via the CLI path — `pnpm run import:pg` writes 1 course / 49 parts / 147 days / 343 activities / 29,426 interactions into dev PG in ~20s. UI path covered in [P1_Manual_Testing.md](P1_Manual_Testing.md) §4.)
- [x] Re-run (same SQLite) → diff shows zero deltas → Import button disabled. (Verified: `pnpm run export:pg && pnpm run diff` after import produces `added=0 removed=0 changed=0 unchanged=343`. Required a small fix in `exportFromPg.ts` to also emit `comment_type` — see [P1_Manual_Testing.md](P1_Manual_Testing.md) §3.)
- [x] Remove an activity in SQLite → re-run → PG row has `deleted=true`, `deleted_at` set, row still exists. **Caveat:** schema has no `deleted_at` column, so we update `updated_at` as the audit timestamp. Row still exists (not physically DELETE'd), which is the critical contractual property.
- [x] `pnpm run export:pg && pnpm run diff` after import produces an empty `diff.json`. (Proved in §3 of manual testing.)
- [x] Post-load sanity checks (Contract Appendix 2 §2.3): — implemented in `src/loader/sanityChecks.ts`, 9 checks total, all pass on the baseline dataset.
  - [x] Row counts per table match expected ranges. (5 checks: courses/parts/days/activities/interactions ≥ 1 active each.)
  - [x] Referential-integrity spot check: every `activity.part_id` resolves to an existing `parts` row; every `interaction.activity_id` resolves; matching-game sub-rows resolve to their activity. (3 raw-SQL `NOT EXISTS` checks on `day_activities`, `activity_interactions`, `part_days`.)
  - [x] Required fields non-null on every row loaded.
  - [x] Report pass/fail per check in the UI result panel. (Rendered under "完整性检查" with ✓/✗ and detail string.)

## Risks

- [x] **Idempotency regression** — covered by a Jest test that runs converter + diff + import twice and asserts the second import is a no-op. → **Covered at runtime by an inline self-check** (`src/loader/selfCheck.ts`): every import, after committing, re-runs `exportFromPg` against the just-written PG and `diffActivities` against the in-memory `data.json`, and appends a pass/fail row to the sanity list. On failure: `ok: false`, CLI exits non-zero, UI panel shows red with 5 divergent URLs. Verified end-to-end by deliberately breaking `exportFromPg.ts` and watching it catch. The automated Jest version (needs a disposable PG fixture) belongs in P2.
- [ ] **OSS credential gaps** — surfaced on first prod-target dry run. If creds missing, fall back to "skip upload, log warning" so the PG load itself isn't blocked. → **N/A for P1** since actual uploads were deferred to P5. Will re-apply when §2 uploads are implemented.
- [x] **Transaction size** — large courses could blow out Prisma's interactive transaction timeout. Fallback: per-Day transactions if a course is > ~500 activities. → **Partially addressed**: timeout bumped to 5 minutes (vs Prisma default 5s) in `load.ts`; first import of 343 activities + 29k interactions completes in ~20s so per-Day fallback is not needed yet. Per-Day fallback logic is **not implemented** — add when a course actually exceeds the timeout.
