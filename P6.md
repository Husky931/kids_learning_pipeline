# P6 — Incremental importer + deploy

## Goal

Importer is fast on re-runs (content-hashed skip path), resumable on failure, and operator-runnable by a non-dev. Full stack is deployed to client's server; Contract §6 Joint Acceptance flow works end-to-end.

## Prerequisites

- [ ] P1 loader exists.
- [ ] P5 API runs against prod PG on the client's server.
- [O] PG + Redis in prod: dual-mode via Docker Compose profiles. `--profile bundled-db` runs them as containers (dev/throwaway); without the flag they're external Aliyun RDS + Aliyun Redis (real prod). `.env.prod`'s `DATABASE_URL` / `REDIS_URL` are the single toggle. See `DEPLOY.md` for both invocations.
- [ ] Client server + DNS + TLS.

## Steps

### 1. Incremental importer hardening

- [ ] Content-hash per activity: sha256 of normalized JSON. Store in `media_processing_log` or new `content_hash` column on each row. Diff-engine uses it for O(1) "unchanged" detection.
- [ ] Media diff: only re-upload files whose source content hash changed. Reuse `media_processing_log`.
- [ ] Resumable: failed load mid-course leaves a recovery marker (`import_run` table tracking last-good checkpoint). Re-run picks up where it stopped without re-doing completed courses.
- [ ] Partial media processing (transcoding / image-opt / watermarking) runs on changed media only.

### 1a. Scheduled SQLite sync (read-only)

Client confirmed: authoring SQLite is pushed to OSS at `oss://ailanguages/videos/sqlite/{YYYYMMDD}.sqlite`, roughly one file per day (may go faster later). Path root is env-driven, never hardcoded.

**Hard rule (from client): no DB write is ever made without an operator clicking the Import button.** The scheduled worker only fetches + diffs; it never writes.

- [ ] New worker process `yyddp-sync` (long-running Node script, runs as a service in `docker-compose.prod.yml` alongside api/importer) — every hour:
  - [ ] Lists objects under `$SQLITE_OSS_PREFIX` (default `videos/sqlite/`).
  - [ ] Picks the object with the highest `YYYYMMDD` prefix.
  - [ ] If that object's key is newer than the last one processed (tracked in `import_run`), proceeds; else sleeps.
  - [ ] Downloads the SQLite, runs converter → `data.json`.
  - [ ] Runs export → `pg-data.json` from PROD PG.
  - [ ] Runs diff → `diff.json`.
  - [ ] Notifies the operator (webhook / log) that a new diff is ready for review.
- [ ] Polling every hour auto-absorbs any cadence change (1/day today, 3/day later, whatever).
- [ ] No auto-import mode. No `SYNC_AUTO_IMPORT` flag. Ever. Operator review is the only path to a PG write.
- [ ] Crash-safe: each run writes a row to `import_run`; overlapping runs are skipped.
- [ ] Admin UI shows "last sync: N hours ago, M deltas pending review" and surfaces the diff.
- [ ] Env vars: `SQLITE_OSS_PREFIX`, `SQLITE_SYNC_INTERVAL_MINUTES` (default 60).

### 2. Operator runbook

- [ ] New: `yyddp-importer/RUNBOOK.md`. Steps:
  - [ ] Drop the new SQLite file into the mounted volume.
  - [ ] Open admin UI (`https://admin.yyddp.com` or `https://yyddp.com/admin/`).
  - [ ] Click "Refresh diff". Review changes.
  - [ ] Click "Import to dev". Review pass/fail report.
  - [ ] Click "Import to prod" (requires second confirm).
- [ ] Include an error-recovery section: resume from checkpoint, re-run failed course only, rollback a bad import (restore soft-deleted rows).

### 3a. Remove mockup scaffolding

By this point mockup mode has served its purpose (it let the API ship before the DB was ready, and let the frontend team work offline through P0–P5). Past Joint Acceptance it's confusing dead code in the client's handover.

- [ ] Delete the `isTestEnvironment` branch from `yyddp-api/src/container.js`. Services bind directly to their real implementations.
- [ ] Delete all `mock*Service.js` / `Mock*Service.js` files under `yyddp-api/src/` (auth, user, course, playlist, parts, activities).
- [ ] Delete `yyddp-api/src/db/mockups/` if any JSON fixtures remain (most should already be gone from P4).
- [ ] Delete `MOCK_JWTTOKEN`, `MOCK_PHONE`, `MOCK_PHONEOTP`, `MOCK_CAPTCHA` from any `.env.*` template files.
- [ ] Confirm `NODE_ENV=mockup` is no longer a valid runtime mode — grep for `=== 'mockup'` and `=== "mockup"` should return zero hits in `src/`.

### 3. Production deployment

- [ ] Finalize the prod env file (`.env.prod`, gitignored) with: real env, prod OSS, prod Aliyun SMS, prod PG host, prod Redis.
- [ ] Stack runs under `docker-compose.prod.yml` with `restart: unless-stopped` on every service — replaces pm2/systemd. `yyddp-api`, `yyddp-importer`, `yyddp-sync`, and `nginx` are all Docker services. Reboot persistence comes from the Docker daemon + restart policy, no `pm2 startup` needed.
- [ ] Nginx runs as a Docker service inside `docker-compose.prod.yml` (not on the host). Config: `/api/` → `yyddp-api:8888`, `/admin/` → `yyddp-importer:5173`/`:3001`. TLS certs in `nginx/certs/`, provisioned via `nginx/certbot-init.sh` (Let's Encrypt) OR supplied by client.
- [ ] Run initial prod PG migrations: `docker compose exec yyddp-api pnpm prisma migrate deploy` (with `DATABASE_URL` in `.env.prod` pointing at prod PG — internal `postgres:5432` for bundled mode, Aliyun RDS endpoint for real prod).
- [ ] If client has existing prod PG with tables: run `prisma db pull` first, reconcile differences, then migrate.
- [ ] First prod import: dry-run against a staging DB if possible, otherwise gated by the P1 safety rails.

### 4. Handover

- [ ] Tagged release (git tag v1.0.0).
- [ ] `DEPLOY.md` at repo root — documents `docker compose --profile bundled-db --env-file .env.prod -f docker-compose.prod.yml up -d` (dev/throwaway) vs `docker compose --env-file .env.prod -f docker-compose.prod.yml up -d` (real prod, external Aliyun), plus migrate, certbot, smoke-test, and importer-first-run steps.
- [ ] Swagger live at `https://yyddp.com/api/v1/docs`.
- [ ] First-run log artifact: proof the prod PG migration + initial import both succeeded.

## Definition of Done — Joint Acceptance (Contract §6)

- [ ] Client runs `docker compose --env-file .env.prod -f docker-compose.prod.yml up -d` on their prod server (no `--profile bundled-db` → external Aliyun PG + Redis), points DNS.
- [ ] A real user signs up via SMS on a real phone, sees their course list, opens a course, completes an activity.
- [ ] Gold is awarded, next activity unlocks.
- [ ] Operator re-runs the importer next day — only changed content is written.

## Risks

- [ ] **Client-side blockers** — server access, DNS control, firewall rules, TLS cert provisioning. Pre-flight all three at EOD before P5 finishes.
- [ ] **Existing prod PG schema** — if client's prod PG has tables that don't match our Prisma schema, reconciliation could be multi-hour. Mitigation: ask for `\dt` output as early as possible.
- [ ] **TLS** — if client supplies certs late, default to a self-signed cert for the handover demo and swap before go-live.
